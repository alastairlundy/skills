# Review: write-changelog

Evaluated against `writing-great-skills`. Resolutions are recorded in `docs/decisions/DECISIONS-skills-write-changelog-review.md` (D001–D020).

## Things that are good

- **Six-step workflow with a clean separation of concerns.** Project Discovery → Commit Retrieval → Message Transformation → Title Selection → Markdown Construction → Output. Each step has a single responsibility; the boundaries are visible in the headings.
- **Ecosystem-aware sub-project detection.** Listing `.csproj`, `package.json`, `go.mod`, `pom.xml` in Step 1 anchors the discovery to a concrete pattern instead of a vague "look at directories" instruction. A future agent can extend the list without re-thinking the rule.
- **Tiered commit analysis in Step 2.** Conventional Commit prefix → diff analysis → user prompt is a robust fallback chain. The cheapest signal is tried first; the most expensive is the last resort.
- **The Keep-a-Changelog ordering is explicit** in Step 5 (Additions, Modifications, Deprecations, Removals, Bug Fixes, Non Source Code). Anchoring the order to a recognised convention reduces invention.
- **The `Use Emojis: Yes/No` toggle** is well-shaped: it is a one-word input, has a sensible default, and flips a single output behaviour. The validation list checks the toggle, so the rule is enforced.
- **Inputs table is small and inline** (4 rows). It fits the "short templates live inline" convention from `AGENTS.md`.
- **"Validation" section as a checklist** is a sharp, scannable contract. A future agent can run the skill and self-check against the same list.
- **Frontmatter description is verb-first and trigger-shaped.** "Generates an ecosystem-aware, user-facing markdown changelog" — verb, two leading words, then the scope. (The description has two typos, addressed separately.)

## Major flaws

- **Step 1's directory-exclusion list is duplicated across two branches.** The no-project-files branch says "exclude `docs/`, `.github/`, `tests/`" and the ambiguous-structure non-interactive branch repeats the same three directories verbatim. The duplication is a drift risk — a future maintainer will fix one and miss the other. *Resolution: D002 — derive the list from `.gitignore` with `docs/`, `.github/`, `tests/` as explicit carve-outs, and emit the inline list as a fallback when `.gitignore` is absent.*
- **Step 6's overwrite path is undefined when the user refuses.** The current text says "ask the user for permission to overwrite before writing" but does not say what happens on refusal. A user who declines in the middle of a session has no path forward. *Resolution: D003 — output the final markdown to the conversation and surface a one-line offer to name a new path.*
- **Step 1 does not probe for a pre-existing changelog.** If the repo already has a `CHANGELOG.md`, the skill invents a new destination path and the user is left reconciling two files. The probe is cheap; the absence is costly. *Resolution: D007 — first-run probe for `CHANGELOG.md`, `HISTORY.md`, `RELEASES.md`, `docs/changelog.md`; default the destination to the discovered path; three-way choice in Step 6 (overwrite / append / refuse).*
- **Step 2's "User Guidance" tier has no autonomous-mode resolution.** A non-interactive run falls through every tier and lands on "ask the user", which is a no-op. The skill has a mode-cut rule (interactive vs non-interactive in Step 1) but Step 2 does not. *Resolution: D009 — autonomous mode resolves tier 1 (Conventional Commit prefix) deterministically; tier 2 (diff analysis) and tier 3 (user prompt) are the same.*
- **Step 1's sub-projects trigger is implicit.** The skill talks about "sub-projects/packages" but does not say when sub-project sections are introduced. A 1-project-file repo gets sub-project sections; a 0-project-file repo gets directory groupings; the threshold between them is implicit. *Resolution: D012 — sub-projects sections are introduced when 2+ project files are detected at the top level; otherwise Global.*
- **Step 2 has no empty-range guard.** A typo in the prior tag (e.g., `v1.0..v2.0` instead of `v1.0..v2.0`) returns zero commits, and the skill silently produces an empty changelog. The error is invisible to the user. *Resolution: D017 — emit a one-line human-readable explanation and a parseable `[CHANGELOG-MARKER] empty-range` marker; CI wrapper translates the marker to a non-zero exit.*

## Minor issues

- **Step 4's ask-questions fallback is hidden in prose.** The fallback "If the tool is unavailable or the user declines, default to Global" is the kind of rule that an agent forgets under context pressure. It should be a numbered step, not prose, and the default should be visible in the output. *Resolution: D001 — fallback is a numbered step in the workflow, default is surfaced as a `(defaulted)` marker on the section header.* (User note: the original concern was reasonable, but "(2) makes sense" — not really a major issue.)
- **Step 4 presents "Other (specify)" as a fourth option.** A free-form text option invites invented categories that do not fit the three named choices. The skill then has to special-case "Other" everywhere. *Resolution: D004 — drop "Other (specify)"; present exactly three choices (Global, All Packages, All Projects); default to Global when none fits.*
- **[STATUS: Closed false positive — see D005] Inputs table Title Case appears inconsistent.** "Prior Git Tag/Commit" vs "Target Branch/Commit" vs "Destination file" vs "Use Emojis" mixes Title Case and sentence case. *Resolution: D005 — closed as a false positive; the prose is consistent (each row's `Input` cell is the user-supplied label, not a heading). Annotation retained here so future readers do not re-open the branch.*
- **"When Not to Use" mentions "clear history".** The phrase is undefined — clear relative to what? The boundary is fuzzy. *Resolution: D008 — replace with a concrete threshold: "repositories with fewer than 5 commits between the prior and target points, or with no tags in the range".*
- **The Inputs table is at the threshold for inline length (4 rows).** A future edit that adds 3 more rows pushes the table past the "short template" convention. *Resolution: D013 — leave inline; revisit in a follow-up if the table grows.*
- **"Sub-projects" and "Packages" are used as if interchangeable.** Step 1 says "sub-projects/packages"; Step 4 offers "All Packages" as a section title; the relationship between the two terms is implicit. *Resolution: D015 — standardise on "Sub-projects"; add a glossary line at the top of Step 1 that defines the relationship ("Package" = a Sub-project that is BOTH a library AND distributable via a package manager).*
- **Frontmatter description has two typos.** "betweeen" → "between"; "speified" → "specified". The global copy at `~/.agents/skills/write-changelog/SKILL.md` was fixed during the grilling session; the repo copy was not. *Resolution: D018 — apply the same edit to the repo copy.*
- **Step 5's "Non Source Code" category name is ambiguous.** "Non Source Code" reads as a negation rather than a category. Keep-a-Changelog uses "Documentation", which is narrower. *Resolution: D020 — keep "Non Source Code" as the category name and add a one-line scope note that documents the wider scope (documentation, configuration, assets, and any other change that is not source code).*

Today the skill says "split them into the following sub-sections" but does not say how a `package.json` `devDependency` is classified. *Resolution: D010 — deterministic three-category rule with explicit ecosystem mappings for JavaScript, Python, Go, and Java; tie-breaker rule for ambiguous cases (file path, not dependency name).*
2. **[STATUS: Closed — see D014] Inline the dependency-classification rule in `SKILL.md`.** *Resolution: D014 — closed; already resolved by D010 (rule content) and D011 (rule location). The hybrid placement is the explicit choice; a re-open to "inline only" would reverse D011.*
3. **Disclose the dependency-classification rule to a reference file** (`references/dependency-classification.md`) with a load-trigger sentence in `SKILL.md` so the LLM reads it before classifying. *Resolution: D011 — short inline summary (3-row table + tie-breaker line) in `SKILL.md`; reference file is the source of truth; load-trigger sentence uses an active voice that an LLM parses as an action.*
4. **[STATUS: Closed — see D016] Add a first-run probe at the top of Step 1** to detect a pre-existing changelog before naming a destination. *Resolution: D016 — closed; already resolved by D007 (probe in Step 1, three-way choice in Step 6). The probe + three-way choice is the explicit resolution; a re-open to "probe only" would reverse D007 Option 1.*

6. **Document the CI integration contract** in a shared `references/ci-integration.md` covering the `[CHANGELOG-MARKER]` marker vocabulary across all decisions that emit markers (D001, D003, D017, and future branches). *Resolution: D017 — same file as the wrapper contract for the empty-range marker.*
7. **[STATUS: Closed — see D019] Move the emoji list to a small table at the top of Step 5** that maps each emoji to its category. *Resolution: D019 — closed; already resolved by D006 (6-row, 2-column table at the top of Step 5).
