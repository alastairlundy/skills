# grilling — review

## Trigger-ability

Description contains `Use when ...` and `Defer to ...` phrasing. The defer-to language is the trigger-shape equivalent of "Don't use when" but is not literally "Don't use when," which causes the description to drift in style from `ask-questions` / `write-changelog` / `skill-architect` (which all use "Do not use for / when"). Standardise.

## Major issues

- **Waza Eval Suite is incomplete** — `evals/grilling/` contains only `tasks/` (6 task files). It is missing `eval.yaml` (the eval configuration AGENTS.md requires) and `fixtures/`. Every other skill in the repo has a complete suite; this one does not. Either complete the suite or remove the directory and document the deviation.
- **The skill is a thin shell over six reference files** — Step 1 of the Workflow requires loading `references/decision-ledger.md`, `options-format.md`, `recommendation-format.md`, `locked-question-format.md`, `tone-and-output.md`, and `convergence-test.md` before the first user question, and applies them *verbatim*. The SKILL.md is therefore not self-contained: an agent that fails to load the references cannot run the skill. The "abort and report the missing file" rule on line 71 makes the failure mode loud, but the load is still optional in the sense that an agent may *think* it has loaded them. A pre-flight file-existence check would help.
- **The `Dxxx` / `Txxx` record format is referenced in code-implementation-grilling and domain-grilling but defined in `grilling/references/decision-ledger.md`** — the parent skill owns the format, but only the children link to it. A user who has the parent skill loaded but not the children will not know the template fields. Either inline a minimal template in `grilling/SKILL.md` or duplicate the link in both child skills.
  - Status: Closed (false positive) — see DECISIONS-skills-grilling-review.md#D008
- **References to `to-prd` and `to-issues` in Step 6** — these are not present in this repo. The skill's exit paths therefore name tools that may not exist in the calling environment. Either add them as separate skills or rename to generic "Plan / Document / Hand off" exit names and document the tool mappings.
- **The "Decision Ledger" file path is hard-coded** — `docs/decisions/DECISIONS-<repo>-<feature>.md`. Many projects do not have a `docs/` directory or use a different convention. The skill is opinionated about project layout in a way other skills in this repo are not.
- **Tone discipline and banned filler words are listed only in the Validation list** — the workflow body does not show the agent the banned list (it sends the agent to `references/tone-and-output.md` instead). The Validation list is the only place the words themselves appear in `SKILL.md`. An agent that has loaded the reference but is in the middle of a branch transition may not remember to recheck.
  - Status: Closed (no change) — see DECISIONS-skills-grilling-review.md#D011

## Minor issues

- **`- [ ] If any reference file was missing or unreadable, the session aborted and the missing file was reported to the user.`** — this is a yes/no check on a property of the *run*, not the *output*. It is a procedure gate, not a content gate. Mixing the two in a single Validation list is fine but the gate is a pre-condition, not a verification of the work.
- **`The next record is Dxxx + 1`** — implicit zero-padding convention. The reference handles this; the SKILL.md does not. An agent that has loaded only the SKILL.md will produce `D1`, `D2`, `D3` not `D001`, `D002`, `D003`. Cite the reference by name on first use, or inline the convention.
- **"Walk the user through it one question at a time" (Step 3) vs. `gate 1` / `gate 2` checks in the references** — the SKILL.md says "one question at a time" but the references' lock format and convergence test use the word "gate" (and "branch"). The terms are used interchangeably. Pick one.
- **`D004` is mentioned in `write-changelog`'s autonomous-mode rule and `D003` is also referenced** — these are decision-ledger IDs from a *different* skill's review. They appear in `write-changelog`'s SKILL.md body. This is the only place such cross-references exist in the repo. They will rot. Replace with descriptive prose.
- **The skill's "When to Use" bullet list mixes triggers and examples** — "Examples: business strategy pivots, product direction, design choices, process changes, organizational structure, hiring, pricing, marketing positioning, partnership decisions." Examples are not triggers. Split them.
- **`Vague decision` is the trigger and is undefined** — what counts as vague? "A 10% revenue decline" is not vague. "Revenue is going down" is vague. The skill does not say. The description's "ambiguous decision" is also not defined. This is a real ambiguity, given the `domain-grilling` and `code-implementation-grilling` skills both have to *not* fire on vague decisions of other types.
  - Status: Closed (no change) — see DECISIONS-skills-grilling-review.md#D005
- **The locked question format string `For [Dxxx] – [branch name]: pick an option, or provide your answer.` is in the Validation list but not in the body** — the body says "open the first decision branch using the locked question format from `references/locked-question-format.md`." The format itself is not in SKILL.md, so the check `Every question used the locked format ... with the Dxxx and name verbatim.` is a check on what was loaded from the reference, not what was in SKILL.md. Fine, but worth noting that the Validation list is checking the *resolved output*, not the *SKILL.md content*.
- **Step 2 says "scan `docs/decisions/DECISIONS-*.md`"** — implies the directory exists. If it does not, the agent is told to "derive the path" and "confirm with the user." This is fine, but Step 1 should pre-flight: `Test-Path docs/decisions` (or equivalent) before scanning.
  - Status: Closed (folded into T04) — see DECISIONS-skills-grilling-review.md#D015
- **Step 4 "Append a `Dxxx` record" assumes the file is open / writable** — no mention of what to do if the write fails (permissions, race with another agent, etc.). Add an error path.
- **"Open follow-up" is referenced as a phase category but not defined here** — appears in `domain-grilling`. Cross-skill references are the only place this term is used.
- **The skill has no concept of "stop grilling" mid-session** — a user might say "actually, never mind, let's just code it." The skill's exit paths are all post-convergence. Add a "user aborts the session" branch.
- **`Decision Ledger` is given a specific filename pattern but the file format is in a reference** — an agent producing a record has to (a) derive the path, (b) load the template reference, (c) read the existing file, (d) compute the next ID, (e) format, (f) write. Six steps for what is conceptually one operation. Consider an inline "produce a Dxxx record" helper.
  - Status: Deferred — see DECISIONS-skills-grilling-review.md#D016
- **`Apply the formats from those files verbatim throughout the session. Do not paraphrase, abbreviate, or modify the formats.`** — extremely strict. But the Validation list does not actually check for verbatim application; it checks for field presence ("four required fields ... at one sentence per field"). The rule and the check are out of sync. Either tighten the check or relax the rule.
- **"Do not batch the writes" is a strict run-time discipline** — but with no consequence stated. If the agent batches anyway, what fails? Nothing observable. Either make it a hard error (must-write-before-next-question) or relax to "preferred."

## Verdict

The skill is the right shape for a generic Socratic interviewer and the relationship with the two children (`domain-grilling`, `code-implementation-grilling`) is well-modelled. The two biggest issues are the missing `eval.yaml` / `fixtures/` (the only skill in the repo that violates the AGENTS.md eval structure) and the depth of the reference-file dependency (six files must load before the first turn). Consider promoting the most-used reference content into inline sections, completing the eval suite, and standardising the "defer to" trigger language.
