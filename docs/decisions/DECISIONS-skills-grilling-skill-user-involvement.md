# Decision Ledger: grilling-skill user involvement

Repo: `skills`. Feature: `grilling-skill-user-involvement`.

## Session context

- **User involvement is defined by option-deferral, not friction mechanisms.** The LLM does not lead with options. The LLM asks the user how to resolve the problem (open-ended), then translates the user's fuzzy or non-concrete answer into 2–4 concrete natural options. The user picks. If none align, the LLM asks clarifying questions. Friction mechanisms (forced-justification, verbatim rephrasing) are secondary and contingent on whether the LLM Recommendation is retained (a later branch).
- **LLM Recommendation fate is unresolved.** The user is uncertain whether to retain or remove recommendations. The decision affects whether D001's Option 3 (forced-justification on the LLM Recommendation) is in scope at all.

### [D001] — what `User involvement` means in this design

- **Resolved Answer**: "At a minimum (1) is involved. Possibly (3) if we keep the recommendations. The main basis for involvement should be that the LLM doesn't provide options upfront. The LLM asks the user how to resolve the problem and the LLM translates fuzzy intent/lack of concreteness into concrete natural options, the user then is asked to pick which one best aligns with what they wanted - If none apply, user is asked questions to clarify the approach they want."
- **Normalized Requirement**: User involvement shall be defined by option-deferral. The LLM shall not lead with options. The LLM shall first ask the user how to resolve the problem (open-ended prompt), then translate the user's fuzzy or non-concrete answer into 2–4 concrete natural options. The user shall then pick the option that best aligns with their intent. If none of the surfaced options align, the LLM shall ask clarifying questions to narrow the user's intent until concrete options can be surfaced. Active user choice on every branch is the minimum baseline; forced-justification on the LLM Recommendation is contingent on the LLM Recommendation being retained (open branch).
- **Constraints**: The D001 Option 3 forced-justification mechanism is NOT in scope while the LLM Recommendation fate is unresolved. Option 1 (active choice) is the minimum baseline. The "If none apply, user is asked questions" path is a refinement loop, not a one-shot — the LLM keeps asking until concrete options are possible, or the user explicitly closes the branch.

### [D002] — what `concrete natural option` means in this design

- **Resolved Answer**: "Concrete means it must be actionable. However we also want it to retain the user's phrasing as much as possible. Natural options must be substantively and/or semantically different enough as to each be justified as an answer. Natural options are options that: are not contrived or created for the sake of having options, make sense given the context provided, are options that can reasonably be performed (must have a basis in reality), and are each individually defensible and advance the user's aim(s)/objective(s). A hybrid/combination should rarely be the answer to a problem - The options should provide meaningful differences. I'd like to require no more than 4 be presented. If the scope is too broad to only resolve 2 to 4 options then the user should be asked which general or specific scope/direction to take."
- **Normalized Requirement**: A `concrete natural option` shall satisfy all of the following: (1) **actionable** — describes a specific single-step action the user could commit to right now; (2) **phrased in the user's words** — preserves the user's own terminology and phrasing as much as possible, with the LLM paraphrasing rather than inventing; (3) **substantively or semantically different** from the other options in the set such that each could be justified as the answer on its own; (4) **not contrived** — must arise from the user's stated context, not be created for the sake of having options; (5) **contextually sensible** — must make sense given the context the user has provided; (6) **performable** — must have a basis in reality, the user can reasonably perform it; (7) **individually defensible** — each option must stand on its own merits; (8) **aim-advancing** — each option must advance the user's stated aim(s)/objective(s). The LLM shall present at most 4 concrete natural options per branch. If the scope of a branch is too broad to resolve into 2–4 options, the LLM shall ask the user to name the scope or direction to take before presenting options.
- **Constraints**: The "aim-advancing" criterion ties the option set to the user's stated goals; if no option advances the user's aim, none are surfaced and the LLM falls back to asking. "Hybrids should rarely be the answer" is a soft rule — the LLM may surface a hybrid when the user's own answer explicitly described one, but hybrids are not the default synthesis strategy. The "scope too broad" fallback is a meta-question about the branch's scope, not a clarifying question about the user's intent; it precedes option generation. "Preserve user's phrasing" is constrained by "actionable" — if the user's phrasing is not actionable, the LLM must sharpen it but must surface the sharpening to the user rather than silently rewriting. The LLM is the translator, not a generator; if the LLM cannot paraphrase from the user's own answer, the fallback is to ask another clarifying question, not to invent options. The term `concrete natural option` is a candidate glossary term; term resolution is open.

### [D004] — what `fuzzy intent` means and what the LLM does when it cannot translate

- **Resolved Answer**: "Fuzzy intent means Unstated specifics or under constrained."
- **Normalized Requirement**: A user answer shall be classified as `fuzzy intent` when it satisfies EITHER of: (1) **unstated specifics** — the answer omits one or more specifics the LLM needs to translate into a concrete natural option (who, what, when, where, how much, by what mechanism); (2) **under-constrained** — the answer could plausibly mean 5 or more different concrete natural options. When the LLM detects a fuzzy intent, it shall ask a clarifying question targeting the missing specifics or the over-breadth of the answer, before attempting to generate options. The LLM shall not surface options while a fuzzy intent is unresolved.
- **Constraints**: The two conditions are detected independently — a single answer can fail one or both. The clarifying question is targeted to whichever condition the LLM detects, not bundled into a single two-part question. The "5+ different options" threshold is the signal for under-constrained; if the LLM can paraphrase into 2–4 defensible concrete natural options, the answer is concrete, not fuzzy. The over-constrained failure mode (specific answer with no meaningful trade-off space) is NOT classified as fuzzy intent under this rule — it is a separate edge case, logged for a follow-up branch. The term `fuzzy intent` is a candidate glossary term; term resolution is open.

### [D005] — LLM Recommendation fate

The D001 forced-justification mechanism (Option 3) is therefore out of scope — no recommendation, no justification. The LLM workflow shall include an explicit "do not offer unprompted" guardrail. A follow-up branch is required to record the LLM-disciplined-no-pre-suggest rule in the skill workflow explicitly. This decision is to be captured in an ADR (D006 open). The term `LLM Recommendation` in the grilling context refers specifically to the agent's named recommendation block at the end of a branch; it does not refer to general guidance, nudges, or capability descriptions.

### [D007] — shape of the "scope too broad" meta-question

- **Resolved Answer**: "The scope/direction is different since they must be mutually exclusive. I think (2) works but it must be shaped based on what the LLM has identified as the high number of natural options and break those down into 2-3 candidate scopes."
- **Normalized Requirement**: When the LLM detects that a branch's scope is too broad to resolve into 2–4 concrete natural options, it shall construct the scope meta-question as follows: (1) identify the high number of natural options the LLM would have surfaced under the broad scope; (2) cluster those natural options into 2–3 **mutually exclusive** candidate scopes, where selecting one candidate scope excludes the others; (3) present the 2–3 candidate scopes as the scope question's options, marked as candidate scopes (not as the final options) and labelled "for example, but not limited to" so the user can still name a scope the LLM has not clustered. After the user picks or names a scope, the LLM re-enters the normal option-generation path within the chosen scope.
- **Constraints**: Candidate scopes must be mutually exclusive — selecting one excludes the others. The 2–3 candidate scope count is itself a cap mirroring the concrete natural option cap (D002); more than 3 candidate scopes is itself a sign the scope is still too broad. The LLM shall not list the underlying natural options that produced the candidate scopes; the scopes are the only surface. Showing the underlying options would re-introduce the "options upfront" failure mode D001 is built to avoid. The "for example, but not limited to" marker preserves the user's right to name a scope outside the LLM's clustering. If the user names a scope outside the LLM's candidates, the LLM proceeds with that scope, not with the candidates — the LLM's candidates are guidance, not a gate. The LLM shall not invent candidate scopes that are not derivable from natural options the LLM would have surfaced; candidates must be clusters of real options, not invented categories. The criterion "mutually exclusive" is a glossary-relevant term (parallel to "substantively or semantically different" in D002 for options); term resolution is open.

### [D008] — over-constrained edge case

- **Resolved Answer**: "Modified answer based on (2) - We go with (2) but the fallback is: The user is asked if several potentially relevant trade-offs would be acceptable. The trade-offs must not have been rejected/refused by the user previously. If any trade-offs of the surfaced trade-off options are accepted -> Surface the original option and present new options based on the newly accepted trade-offs. If no new trade-offs are accepted -> Fallback to (1)"
- **Normalized Requirement**: When the LLM detects an over-constrained user answer, the LLM shall not surface the single concrete natural option directly. Instead, the LLM shall: (1) identify several (target 2–3) potentially relevant trade-offs the user might accept in exchange for a different choice; (2) filter the trade-off list to exclude any trade-off the user has previously rejected or refused in the same session; (3) ask the user which of the surfaced trade-offs, if any, they would accept. The LLM shall then branch: (A) **if the user accepts one or more trade-offs**, surface the single concrete natural option AND present new concrete natural options constructed around the newly accepted trade-offs (a trade-off acceptance is treated as a new constraint the LLM uses to construct 2–4 new options); (B) **if the user accepts no new trade-offs**, fallback to Option 1 — surface the single concrete natural option and ask the user to confirm it or expand the scope by naming what they would change. The session shall maintain a running record of trade-off rejections so the LLM does not re-offer them.
- **Constraints**: The "several" target is 2–3 trade-offs, mirroring the cap discipline in D002 (concrete natural options) and D007 (candidate scopes). Trade-offs the user has previously rejected in the same session are off-limits — the LLM must track session-level trade-off rejection history. The "accepted trade-offs expand the option set" pattern is the bridge from over-constrained to multi-option: a trade-off acceptance is treated as a new constraint the LLM uses to construct 2–4 new options. The fallback (no new trade-offs accepted → surface single option → user confirms or expands scope) is a terminal state, not an infinite loop; the LLM does not ask trade-off questions indefinitely. The trade-off question is a distinct question type — separate from the natural-options question, the scope meta-question (D007), and the clarifying question (D004). The session-level rejection history lives in the agent's context (conversation memory), not in a separate file. The fallback to Option 1 inherits Option 1's risk that the user rubber-stamps the single option; the over-constrained path's purpose is to mitigate that risk by exhausting the trade-off space first. The terms `trade-off` (as a question type), `session-level trade-off rejection history`, and `trade-off acceptance` are glossary candidates; term resolution is open.

### [D009] — shape of the open-ended branch-starting prompt

- **Resolved Answer**: "(1) but I'm not satisfied with the phrasing 'What would you want to do on that' also the '-' in the sentence should be replaced with a comma."
- **Normalized Requirement**: The open-ended branch-starting prompt shall have a two-part structure: (A) the LLM's paraphrase of the user's aim, in the LLM's own voice (not verbatim, not word-for-word, to avoid the LLM's framing being mistaken for the user's words); (B) a dimension-led open question that names the branch and the dimension of the user's thinking the LLM is asking about, with a **comma** (not an em-dash) connecting the dimension question to a follow-on "what would 'good' look like for you there?". The full prompt template is: "Within that, the part of the work is <branch name>. What's your thinking on <dimension>, and what would 'good' look like for you there?" The dimension is drawn from the branch's substance, not invented. The LLM must be willing to use a generic dimension (e.g., "how do you want this named") when the branch has no clean dimension, rather than contort the branch to fit the prompt structure.
- **Constraints**: The em-dash is forbidden in the prompt template; the comma is required. The "what's your thinking on" phrasing replaces the rejected "what would you want to do on that" phrasing because the user found the latter too action-led and vague about the answer shape. The "what would good look like for you there" follow-on is the non-terse answer shape the user requested; it gives the user room to express their reasoning without leading the conclusion. The dimension is not given as an example ("you might think about <dimension>") because Option 3's example-dimension risk was rejected; the dimension is named directly. The dimension naming is itself a translation step on the LLM. The "not verbatim" rule on the aim paraphrase prevents the LLM's framing from being mistaken for the user's words. The aim paraphrase is one sentence, not multiple, to avoid the restatement becoming noise across many branches. The prompt template is a starting point, not a strict string — the LLM may substitute a more natural phrasing that preserves the two-part structure and the comma rule, but the "what's your thinking on <dimension>, and what would 'good' look like for you there" shape is the canonical form for the option to be recognisable across branches.
