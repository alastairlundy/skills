# Decision Ledger — skills / spec-to-tickets-ticket-sizing

Topic: revise the `spec-to-tickets` skill so tickets are a sane size and scope for both human and AI-agent implementers, and the path from ticket creation to PR filing is clear. Surface the AI-vs-human split as a possible outcome.

### [D001] — goals for the session

- **Resolved Answer**: Two goals. Goal 1: tickets should be a sane size and scope for both humans and agents — not so big that a human needs weeks, not so small that context-switching overhead dominates; the size must respect the human's cognitive load and mental-model switching, and the agent's context size and agentic persistence. Goal 2: the path from creating tickets to filing a PR should be clear — the current translation from tickets to PR(s) is not well-defined, and the skill must support both single-PR and multi-PR cases. Two side notes (not goals): (a) parallelizing changes to the same file in different tickets is generally ill-advised regardless of implementer; (b) exact time/effort prescription in ticket content probably does not matter — relative size may help ground implementers, but over-prescription creates false expectations.
- **Normalized Requirement**: The `spec-to-tickets` skill shall produce tickets whose size and scope are workable for both human and AI-agent implementers, with a clear path from ticket creation to PR filing that supports both single-PR and multi-PR outcomes. The skill shall discourage parallel execution of tickets that touch the same file. The skill shall avoid over-prescribing exact effort values in ticket content while permitting relative-size labels.
- **Constraints**: Human and agent sizing drivers differ — humans depend on cognitive load and mental-model switching, agents depend on context size and agentic persistence. The PR path may sometimes map one decision ledger or implementation document to multiple PRs; the skill must support that. Exact-effort prescription in ticket content is forbidden; relative-size labels are allowed.

### [D002] — sizing primitive and implementer-type split

- **Resolved Answer**: "Option 1 - Replace time/effort with a scope/coherence primitive"
- **Normalized Requirement**: The `spec-to-tickets` skill shall size tickets by what they coherently cover (one logical concern, one reviewable unit, one mergeable change), not by time or effort; the same primitive shall apply to both human and AI-agent implementers, producing a single ticket shape.
- **Constraints**: The new primitive requires a precise definition (what counts as "coherent"?) — that definition is a follow-up branch. Time-based size labels (XS = under 1 hour, etc.) are removed. The original two-paths temptation (separate AI and Human ticket shapes) is rejected; any AI-vs-Human differences that survive belong in PR-path mechanics, not in ticket sizing.

### [D003] — PR path timing

- **Resolved Answer**: "Option 1 makes sense given the current situation but it may be worth revisiting in the future."
- **Normalized Requirement**: The `spec-to-tickets` skill shall ask the user (or read from the spec) for the anticipated PR count at the start of the workflow; if unspecified, all tickets shall be grouped under a single PR.
- **Constraints**: The up-front ask is provisional — the user has flagged that the decision may be revisited. The override mechanism (user, spec, ticket) is a follow-up branch.

### [D004] — PR-path override mechanism

- **Resolved Answer**: "Option 2 narrowed to require an explicit call for multiple PRs to be a definitive answer; non-explicit calls are flagged only if they conflict."
- **Normalized Requirement**: The `spec-to-tickets` skill shall accept three override sources for the PR count, in priority order — User (1st), Spec (2nd), Supporting Documentation (3rd); only an explicit call for multiple PRs shall be treated as a definitive source answer; non-explicit calls (contemplates, could, might, etc.) shall be weak signals that trigger a reconciliation ask to the User only when they conflict with a higher-priority source's answer; the ticket is not an override source.
- **Constraints**: `Supersedes: D003` (D003's constraint referenced "user, spec, ticket" as override sources — the ticket is removed and Supporting Documentation is added). Supporting Documentation is any document or file provided (besides the spec) or referenced that contributes to understanding the work. "Explicit" is the threshold for a definitive source answer; the LLM must distinguish explicit from non-explicit language in source text. The reconciliation ask is a User action, not an LLM action.

### [D005] — coherence primitive

- **Resolved Answer**: Coherence has two levels. Ticket-set coherence: all tickets collectively work toward a shared objective; different tickets can focus on different units of work, types of work, or themes, but they collectively must work toward a shared objective. Per-ticket coherence: each ticket must satisfy three sub-criteria — (1) a single mergeable change: a set of individual changes that collectively and coherently become a single change to be made in the codebase (e.g., implementing a new feature); (2) a reviewable unit: a unit of work, such as a change or set of changes, that can be logically identified as belonging together, and that collectively address an issue in a way that they would individually or separately not be able to; (3) a single logical concern: an area of work that focuses on dealing with a specific domain or problem.
- **Normalized Requirement**: The `spec-to-tickets` skill shall define coherence as: (a) all tickets in a set collectively work toward a shared objective; (b) each ticket is a single mergeable change, a reviewable unit, and addresses a single logical concern. A ticket that fails any sub-criterion is incoherent and must be split or rescoped before publication.
- **Constraints**: The three sub-criteria are not fully independent — a single mergeable change typically produces a reviewable unit, and a single logical concern typically produces a single mergeable change — but the LLM must apply all three explicitly. The "shared objective" applies per ticket set; in a multi-PR spec, each PR's ticket set has its own shared objective (per D003-D004). "Coherent" is the meta-rule; the three sub-criteria are its operationalization.

### [D006] — relative-size labels

- **Resolved Answer**: The existing labels (XS/S/M/L/XL) are removed. Size is signaled by file count and line count, with no duration prescription. For tickets that involve files: present the number of files to create, edit, or delete (as explicitly described by the ticket); use the language "Large Files to be created" if any new file is >= 500 lines; use the language "Large Edits required" if the total lines to add, remove, or change is >= 500 lines. For tickets that do not involve any file additions, edits, or deletions: no relative size is offered.
- **Normalized Requirement**: The `spec-to-tickets` skill shall remove the existing size labels (XS/S/M/L/XL) and shall signal relative ticket size as follows: for tickets that involve files, the skill shall present the number of files to create, edit, or delete (as explicitly described by the ticket) and shall apply the labels "Large Files to be created" (when any new file is >= 500 lines) and "Large Edits required" (when the total lines to add, remove, or change is >= 500 lines); for tickets that involve no file changes, the skill shall offer no relative size.
- **Constraints**: The 500-line threshold is a fixed number, not a relative signal. The "Large" language is descriptive, not prescriptive — it does not imply duration or effort. The file count is the number of files explicitly described by the ticket, not inferred from adjacent code. Tickets that involve only documentation, decisions, or other non-file artifacts have no relative size signal.

### [D007] — parallelization rule for same-file changes

- **Resolved Answer**: When a ticket modifies a named specific file, later tickets that modify that same named file shall be blocked by the initial ticket until it is completed. The existing `blocked by` field is the mechanism: the LLM shall automatically populate `blocked by` with the earlier ticket whenever a later ticket's file list overlaps with the earlier ticket's file list. The rule applies regardless of implementer (human or AI agent).
- **Normalized Requirement**: The `spec-to-tickets` skill shall automatically populate the `blocked by` field for any ticket whose file list overlaps with another ticket's file list, such that the later ticket is blocked by the earlier ticket. The user may override the LLM's automatic `blocked by` population if they want different ordering.
- **Constraints**: When both tickets are leaves (neither is blocked by the other from elsewhere), the LLM must pick a deterministic ordering (e.g., the ticket listed first in the proposal is treated as the earlier one). The rule applies to any file listed in the ticket's file scope; partial overlap is sufficient to trigger the block. The file scope is the set of files explicitly described by the ticket (consistent with D006's "as explicitly described by the ticket" rule).

### [D008] — workflow integration

- **Resolved Answer**: Concrete mapping: D003 → Step 5 (end, after output target, mode-specific per D009); D004 → Step 6 (start, before 6.1, mode-specific per D009); D005 → Step 6.2 (during proposal, validation gate, same in both modes); D006 → Step 8 (during ticket body generation, same in both modes); D007 → Step 8 (after ticket bodies, before finalization, same in both modes).
- **Normalized Requirement**: The `spec-to-tickets` skill's existing 10-step workflow shall absorb the new rules as follows: the PR count ask (D003) is added at the end of Step 5 (Output Target Resolution); the override mechanism (D004) is added at the start of Step 6 (Ticket Decomposition Proposal), before 6.1; the coherence check (D005) is added inside Step 6.2 (Ticket Proposal) as a validation gate during proposal; the relative-size signal (D006) is added inside Step 8 (Ticket Generation) during ticket body generation; the parallelization `blocked by` rule (D007) is added inside Step 8 (Ticket Generation) after ticket bodies are generated and before the dependency graph is finalized.
- **Constraints**: D003 and D004 have mode-specific behavior per D009; the other rules apply identically in Collaborative and Self-Contained modes. The validation gate in D005 means a ticket that fails any sub-criterion (mergeable change, reviewable unit, logical concern) is rescoped or split before being accepted into the proposal. The parallelization rule in D007 requires the file list to be known, which is established during ticket body generation in Step 8.

### [D009] — Self-Contained mode for PR count and override mechanism (re-open of D003, D004)

- **Resolved Answer**: "Option 1 - Spec is the User's voice in Self-Contained mode; read the spec for PR count signals, default to 1 PR if none; no reconciliation ask in Self-Contained mode."
- **Normalized Requirement**: In Self-Contained mode, the `spec-to-tickets` skill shall treat the spec as the User's voice for PR count purposes: the skill shall read the spec for explicit PR count signals, default to 1 PR if no signal is found, and shall not issue reconciliation asks (because no user is in the loop). In Collaborative mode, the D003 and D004 rules apply as previously recorded.
- **Constraints**: `Supersedes: D003, D004` (the Self-Contained portion of D003 and D004 is now defined by D009; the Collaborative portions remain in force). The "spec as User's voice" rule preserves the priority order by substitution: User (Collaborative) or Spec (Self-Contained) > Spec > Supporting Documentation. Specs that don't mention PR count will default to 1 PR in Self-Contained mode; the user must include the PR count in the spec to override.
